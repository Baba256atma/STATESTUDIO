# Nexora Investor / Customer Demo Script

This script is based on the real current Nexora MVP.

Current product truth:
- One shared Nexora core engine
- Multiple domain experiences
- Launch domains: Business, DevOps, Finance
- Real current flow: domain selection -> workspace -> starter demo -> prompt -> scene update -> risk / fragility -> executive response

Target audience:
- Investors
- Consulting firms
- Strategy teams
- CTO / DevOps leaders
- Finance analysts

Target length:
- 5 to 7 minutes

## Demo Goal

The demo should land five points:

1. Nexora models systems, not dashboards.
2. Nexora reveals fragility and risk propagation.
3. The same engine works across multiple domains.
4. Nexora can support scenario exploration.
5. Nexora produces executive-level insight, not only technical output.

## Demo Setup

Use the current MVP exactly as it exists:
- Start on the domain selection screen.
- Use the three launch domains only: Business, DevOps, Finance.
- Let each domain auto-load its default starter demo.
- Use short prompts from the visible prompt examples.
- Show the executive dashboard and risk/fragility views as part of the main story.

Avoid in this demo:
- Preview domains as primary proof points
- Future multi-source scanning as if it is already fully shipped
- Autonomous exploration as if it is already a polished user-facing workflow

## 1. Introduction

Time:
- 30 seconds

Presenter says:
- "Nexora is not a dashboarding tool. It is a system-intelligence platform."
- "It models how pressure moves through a system, where fragility appears, how risk propagates, and what action makes the most sense next."
- "The key idea is one shared engine, with multiple domain experiences on top of it."

Presenter action:
- Open Nexora at the domain selection screen.

Audience sees:
- A domain selection view with Business, DevOps, and Finance marked as launch domains.

Insight revealed:
- Nexora is presented as one platform, not separate point solutions.

Transition:
- "Let me show that first at the product level."

## 2. Domain Selection

Time:
- 30 seconds

Presenter says:
- "The same core engine is underneath every domain."
- "What changes is the experience layer: the starter demo, prompt vocabulary, panel emphasis, and executive framing."

Presenter action:
- Click `Business`.
- Briefly point out the prompt examples and selected-domain preview.

Audience sees:
- Business selected as a launch domain.
- Business-specific prompt examples and default demo label.

Insight revealed:
- Nexora is domain-aware without being a different product each time.

Transition:
- "We’ll start with business systems, then move to technical systems, then financial systems, all on the same core."

## 3. Business Domain Demo

Time:
- 90 seconds

Presenter says:
- "In the Business domain, Nexora models a live operating system, not a static org chart."
- "This demo starts with a business fragility map across suppliers, operations, inventory, fulfillment, demand, cash pressure, and customer trust."

Presenter action:
- Enter the Business workspace.
- If the scene has not already loaded, click `Start Business Demo`.
- Let the starter scene settle.
- Use one visible prompt chip or type: `supplier delay`.

Audience sees:
- A business system map with highlighted focus objects.
- A domain-specific prompt guide.
- After the prompt, the scene updates and the right-side analysis surfaces populate.

Insight revealed:
- Business pressure is modeled as a system with dependencies and cascading effects, not as isolated KPIs.

Presenter says:
- "What matters here is not only the supplier issue. It’s how pressure moves into operations, inventory, fulfillment, and trust."
- "Nexora makes the chain of consequences visible."

Presenter action:
- Open or point to `Risk Flow`.
- Then open or point to the `Executive Dashboard`.

Audience sees:
- Fragility / risk propagation context.
- Executive framing that explains what happened, why it matters, and what action is recommended.

Insight revealed:
- Nexora turns operational pressure into decision-level guidance.

Transition:
- "This is not only a business tool. The same engine can model technical systems."

## 4. DevOps Domain Demo

Time:
- 90 seconds

Presenter says:
- "Now we switch domains, but not engines."
- "In DevOps, the same system model becomes a service dependency and reliability model."

Presenter action:
- Use `Change Domain`.
- Select `DevOps`.
- Enter the DevOps workspace.
- Let the default DevOps starter demo load.
- Use one visible prompt chip or type: `database latency`.

Audience sees:
- A technical system map with API Gateway, Auth Service, Database, Queue, Worker Pool, Cache Layer, Latency Pressure, Error Rate, and Service Reliability.
- The prompt updates the scene and analysis outputs.

Insight revealed:
- The engine is general enough to handle technical dependency analysis without becoming a separate product.

Presenter says:
- "Here the problem is service fragility. Latency in one dependency can propagate through the queue, worker path, and eventually into reliability."
- "What you’re looking at is failure propagation across a live system model."

Presenter action:
- Point to `Risk Flow`.
- Point to the executive or advice view.

Audience sees:
- Dependency-driven propagation and recommended stabilizing action.

Insight revealed:
- Nexora is useful for engineering leaders because it translates technical instability into operational consequences and next steps.

Transition:
- "And the same engine can shift again into financial systems."

## 5. Finance Domain Demo

Time:
- 90 seconds

Presenter says:
- "In Finance, the same core model becomes a market-fragility and liquidity-stress model."
- "This starter workspace focuses on liquidity, exposure, leverage, volatility, credit pressure, and capital stability."

Presenter action:
- Use `Change Domain`.
- Select `Finance`.
- Enter the Finance workspace.
- Let the default finance demo load.
- Use one visible prompt chip or type: `liquidity stress`.

Audience sees:
- A finance system risk map with market demand, asset price, liquidity, portfolio exposure, leverage, credit pressure, volatility, and capital stability.
- The prompt updates the scene and risk surfaces.

Insight revealed:
- Nexora can represent financial fragility with the same shared engine used for business and DevOps.

Presenter says:
- "This is not a dashboard of disconnected financial metrics."
- "Nexora is showing how funding pressure, volatility, and exposure concentration can propagate into systemic fragility."

Presenter action:
- Point to fragility / risk flow.
- Point to executive summary / action framing.

Audience sees:
- A readable financial risk story and a stabilization-oriented recommendation.

Insight revealed:
- Finance users get a system-level view of downside risk, not just a static report.

Transition:
- "At this point, the pattern should be clear: one engine, three different domain experiences."

## 6. Prompt Interaction and System Response

Time:
- 45 seconds

Presenter says:
- "The important interaction in Nexora is not filling out forms. It’s expressing a system pressure in plain language."
- "A short prompt updates the active system, reveals the hot path, and changes the decision surface."

Presenter action:
- Stay in the current domain.
- Enter one more short prompt if needed, such as `market volatility spike` or `traffic spike`, depending on the audience.

Audience sees:
- Prompt -> system update -> scene changes -> analysis refresh.

Insight revealed:
- Nexora is interactive system analysis, not static reporting.

## 7. Risk Propagation

Time:
- 30 seconds

Presenter says:
- "What Nexora does especially well is make propagation visible."
- "The system does not only identify what is weak. It shows how weakness travels."

Presenter action:
- Keep the `Risk Flow` or fragility-focused panel visible.

Audience sees:
- The path of pressure moving through connected objects and outcomes.

Insight revealed:
- The product value is the causal system view, not just a score.

## 8. Executive Insight

Time:
- 45 seconds

Presenter says:
- "And this is where Nexora becomes decision software."
- "It does not stop at detecting instability. It turns the active system state into an executive brief."
- "That means: what changed, why it matters, and what stabilizing action to take next."

Presenter action:
- Show the `Executive Dashboard`.

Audience sees:
- A readable executive summary grounded in the current system state.

Insight revealed:
- Nexora is usable beyond analysts. It can support executives, consulting teams, and domain leads.

Transition:
- "Today this is already usable in the MVP. The roadmap extends that same engine even further."

## 9. Scenario Exploration Preview

Time:
- 30 seconds

Presenter says:
- "The next layer of Nexora is not a separate product. It is the same system model used for deterministic scenario exploration."
- "That means generating candidate stress cases, simulating them, comparing outcomes, and surfacing the most dangerous or stabilizing futures."
- "That capability is already aligned architecturally, but the MVP demo today focuses on the current prompt-driven product path."

Presenter action:
- Do not navigate into non-MVP UI that does not exist.
- Keep this verbal.

Audience sees:
- The existing workspace, while the presenter explains the near-term platform extension.

Insight revealed:
- There is credible product depth beyond the current MVP, without pretending unfinished UI already exists.

## 10. Final Platform Message

Time:
- 30 seconds

Presenter says:
- "Nexora is a new kind of software category: system intelligence."
- "It models how pressure moves through business systems, technical systems, and financial systems."
- "It reveals fragility, shows propagation, and helps teams decide what to do next."
- "What matters is that this is one shared engine, not a collection of disconnected domain tools."

Presenter action:
- Pause on the active workspace with the executive view visible.

Audience sees:
- A domain-aware Nexora workspace with system map plus executive framing.

Insight revealed:
- Nexora is positioned as a platform for system modeling, fragility discovery, and decision support.

## Short Demo Path

Use this if you only have 3 to 4 minutes:

1. Show domain selection and explain one engine, multiple domains.
2. Run the Business demo with `supplier delay`.
3. Switch to DevOps and run `database latency`.
4. Switch to Finance and run `liquidity stress`.
5. End on executive insight and the platform vision.

## Suggested Prompt Choices

Business:
- `supplier delay`
- `demand spike`
- `cash pressure`

DevOps:
- `database latency`
- `queue backlog`
- `service dependency failure`

Finance:
- `liquidity stress`
- `market volatility spike`
- `credit pressure rising`

## Presenter Notes

- Keep the demo moving. Do not explain every panel.
- Narrate propagation, not only object names.
- Always translate what the audience sees into "why this matters."
- If the audience is technical, spend more time in DevOps risk flow.
- If the audience is investor or consulting-heavy, spend more time on Business and the final platform message.
- If the audience is finance-oriented, give Finance the longest prompt segment.
